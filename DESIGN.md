# Hearth Design System

Hearth is the Utility Warehouse design system — a monorepo publishing React components, design tokens, icons, and illustration assets under the `@utilitywarehouse/` npm scope.

**Packages**
- `@utilitywarehouse/hearth-react` — React component library (`packages/react`)
- `@utilitywarehouse/hearth-tokens` — Design tokens: CSS variables + JS exports (`packages/tokens`)
- `@utilitywarehouse/hearth-react-icons` — 194 React SVG icon components (`packages/react-icons`)
- `@utilitywarehouse/hearth-svg-assets` — SVG illustration assets (`packages/svg-assets`)

All CSS tokens are prefixed `--h-*`. Always use semantic tokens (`--h-*`) in UI code; never use raw primitive `color.*` values directly.

---

## Overview

Hearth is a utility-first design system built for the Utility Warehouse multi-product platform. The visual language is warm, approachable, and purposeful — rooted in a deep brand purple with warm-white surfaces that feel human rather than sterile.

Six UW service lines — **energy**, **mobile**, **broadband**, **insurance**, **cashback**, and **pig** (savings) — each carry a distinct colour. These service colours appear consistently across components, tokens, mascot illustrations, and product scenes, making product context visually immediate. Every component surface, shadow, and icon can reflect the service being presented.

The type system pairs a custom display face (Comic Hams) for headings with DM Sans for body copy and DM Mono for detail/label contexts. The overall rhythm is generous: warm-white backgrounds, large border-radius, and soft coloured shadows create a layered but uncluttered hierarchy.

## Do's and Don'ts

- **Do** use semantic tokens (`var(--h-*)`) in all UI code — never hardcode raw hex values from `color.*`
- **Do** wrap your app in `<HearthProvider>` before using any Hearth component
- **Do** use the `inverted` prop on text and button components when placing them on brand-purple or service-coloured surfaces
- **Do** match the `-dark` / `-light` SVG asset variant to the background surface
- **Do** use the `title` prop on icons when they convey meaning (not just decoration)
- **Do** use `--h-focus-primary` on light surfaces and `--h-focus-inverted` on dark surfaces for focus rings
- **Do** apply `--h-opacity-disabled` (0.5) universally for disabled states — do not customise per-component

- **Don't** use raw `color.*` primitive values in UI code — always go through semantic tokens
- **Don't** mix the six service colour names (energy, mobile, broadband, insurance, cashback, pig) with generic colour descriptions — always use the vocabulary name
- **Don't** use `asChild` and `as` props together on the same component
- **Don't** place more than one `HearthProvider` in the tree
- **Don't** use WCAG-failing colour combinations — check any custom `color`/`backgroundColor` pair achieves at least 4.5:1 contrast for normal text
